## 🎯 實作要點 / 設計決策 / Implementation notes
1. **Framed root cause at the git-level, not the workflow-level**
   - Why: The logical isolation ("each worker edits their own row") was correctly followed — the failure is that git stages at file level, not line level. Blaming the workflow convention would lead to more process; blaming the physical model leads to the correct fix (worktrees).
   - Alternative considered: Recommending `git add -p` (patch mode) as a mitigation — rejected because it's interactive and error-prone under automation.

2. **Status set to `resolved` with forward references to M6.2–M6.4**
   - Why: The fix (worktree-per-worker) is being implemented in this same wave. Setting `fix-pending` would require a follow-up edit; `resolved` with "implementation: M6.2–M6.4" is accurate since the design decision is made.
