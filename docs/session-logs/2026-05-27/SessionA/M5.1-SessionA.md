## 🎯 實作要點 / 設計決策 / Implementation notes
1. **New step 2: load codebase-memory tools early, flag availability**
   - Why: Later steps (6a, 9a) need to know whether codebase-memory is available. Loading once at the start avoids repeated ToolSearch calls and gives a clean boolean flag.
   - Alternative considered: Inline ToolSearch at each usage point — rejected because it adds redundant calls and complicates each step's logic.

2. **Step 6a: trace_path for hidden dependency detection (advisory only)**
   - Why: File-region partitioning only checks explicit file lists in PROGRESS.md. `trace_path` can discover callers/importers outside the milestone scope — the #1 cause of cross-Worker conflicts that static file lists miss.
   - Design choice: Advisory (⚠️), not blocking. The Reviewer decides whether to adjust scope. Blocking would be too aggressive given that codebase-memory's graph may be stale or incomplete.

3. **Step 9a: search_graph to enrich technical hints**
   - Why: Better hints = faster Worker onboarding. Graph context (imports, callers, architectural role) gives Workers actionable starting points instead of "read the file and figure it out."
   - Design choice: Separate sub-section (9a) before the message structure (9b) to keep the existing message template clean. Graph findings are woven into the hints field, not added as a new dispatch section.

4. **Renumbering: 2-10 → 3-11**
   - Why: Inserting step 2 (load tools) requires shifting all subsequent step numbers. Cross-references within the file (e.g. "loop back to step 5") updated to match.
