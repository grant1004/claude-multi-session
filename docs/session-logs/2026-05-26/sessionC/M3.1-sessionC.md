# M3.1 — Add `/multi-session:dispatch` helper command

> Commit `f3e44b9` · sessionC · 2026-05-26

## 📦 變更摘要 / Change summary
- `plugins/claude-multi-session/commands/multi-session/dispatch.md`: New slash command — 10-step workflow guiding Reviewer through milestone selection, worker assignment, conflict detection, and dispatch message generation

## 🎯 實作要點 / 設計決策 / Implementation notes
1. **Read-only command with no send_message capability**
   - Why: The dispatch step is the highest-stakes action in the workflow (wrong "don't touch" list = git conflicts). Forcing Reviewer to review + manually send catches errors.
   - Alternative considered: auto-send with confirmation prompt — rejected because it shifts the default from "review carefully" to "rubber stamp"

2. **File-region conflict detection as a first-class step (step 5)**
   - Why: This is the most error-prone part of manual dispatching. With 3+ workers, mentally cross-referencing file lists is tedious and failure-prone.
   - Approach: Compare chosen milestone's expected files against ALL in-flight milestones' file sets. If overlap, force an explicit acknowledge before proceeding.

3. **First-dispatch detection via worker summary heuristic (step 7)**
   - Why: The onboarding pre-block is long and cluttering it into every dispatch wastes worker context. But omitting it on first dispatch leaves the worker unprimed.
   - Heuristic: summary contains "awaiting"/"standing by" → first dispatch; contains "working on"/milestone ID → subsequent. Final decision exposed to Reviewer via AskUserQuestion.

4. **Worktree info auto-derivation from git worktree list (step 7)**
   - Why: Workers need exact worktree path + branch in the dispatch. Deriving from git state prevents typos.
   - Fallback: If worktree not found, output a creation command hint for the Reviewer.

5. **Adaptive build rule in generated message (step 8)**
   - Why: The dispatch template's rule 2 must match the project's actual build step. Pure-markdown projects say "verify YAML frontmatter"; npm projects say "npm run build".
   - Approach: Detect from project manifest presence or PROGRESS.md audit summary.
