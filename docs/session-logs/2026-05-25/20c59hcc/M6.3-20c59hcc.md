## 🎯 實作要點 / 設計決策 / Implementation notes
1. **Worktree verification as step 0 (not a separate pre-block)**
   - Why: The dispatch already has a first-dispatch pre-block for onboarding. Adding a separate "worktree setup pre-block" would create two conditional blocks. Instead, step 0 was prepended to the existing pre-block and the worktree path announcement was added as a standalone line. Simpler for the Reviewer to compose.
   - Alternative considered: Separate "Worktree setup pre-block" section — rejected because it duplicates the conditional-inclusion pattern and every first dispatch already needs the worktree info.

2. **Merge-first sequence in review-pass.md**
   - Why: M6.2's reviewer.md already mandates `git merge --ff-only` on pass. The review-pass template's "after-pass actions" is the checklist Reviewers actually follow, so it must match. Merge goes first (step 1) because PROGRESS.md edits should happen on main after the merge lands.

3. **Rules count bumped from 4 to 5 in dispatch template**
   - Why: The branch commit rule is non-negotiable (committing to main bypasses review). Making it an explicit numbered rule rather than burying it in hints ensures Workers don't miss it.
